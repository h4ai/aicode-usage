> 状态：✅ 全部确认，可开发 | 作者：OPS | 日期：2026-04-17

## 一、项目背景

企业内部部署 AI 编程助手（IDE 插件），客户端上报使用事件到 ClickHouse（`otel.events`）。
本系统为每位用户提供个人 Token 使用看板，通过企业 AD 域认证，数据按用户隔离。

---

## 二、用户角色 ✅

| 角色 | 说明 | 认证方式 |
|------|------|----------|
| **普通用户** | AD 域账号登录，只看自己的数据 | 企业 AD（Windows 标准 LDAP，389端口） |
| **管理员** | 写死在配置文件中，支持多账号 | 用户名 + bcrypt hash 密码（修改配置文件即时生效） |

---

## 三、认证模块 ✅

- 普通用户：企业 Windows AD（LDAP，地址在 config.yaml 中配置）
- 登录匹配：姓名 **或** 域账号，任意一个与 ClickHouse `userId` 字段匹配即可
- `userId` 格式：`zhangsan`（域账号格式）
- 管理员：config.yaml 中配置，支持多账号，密码存储 bcrypt hash
- 登录成功颁发 JWT，有效期 8 小时
- 修改管理员密码后，旧 JWT 立即失效
- 后端所有查询强制 `WHERE userId = 当前用户`（管理员不受限）

## 四、配额体系 ✅

### 4.1 配额级别

- 初始预设 **L1 / L2 / L3** 三个级别，管理员可在界面中灵活调整
- 用户未分配级别时，默认使用 **L1**
- 用户首次 AD 登录后自动创建用户记录，默认分配 L1
- **配额级别固定三级（L1/L2/L3），不支持删除**，管理员只能修改各级别的额度数值

### 4.2 双重控制

| 维度 | 周期 | 超出后处理 |
|------|------|-----------|
| 月度 Token | 自然月（服务器时区，每月1号 0点重置） | 仅告警，不拦截 |
| 每日请求次数 | 自然日（服务器时区，每天 0点重置） | 仅告警，不拦截 |

### 4.3 告警展示

**月度 Token 进度条：**

| 使用比例 | 颜色 | 文案 |
|----------|------|------|
| < 50% | 🟢 绿色 | 使用正常 |
| 50%~79% | 🟡 黄色 | 已使用 {x}%，请注意控制用量 |
| 80%~99% | 🟠 橙色 | 已使用 {x}%，即将达到上限 |
| ≥ 100% | 🔴 红色 | 已超出月度限额，请联系管理员 |

**每日请求次数进度条：**

| 使用比例 | 颜色 | 文案 |
|----------|------|------|
| < 80% | 🟢 绿色 | 今日使用正常 |
| 80%~99% | 🟠 橙色 | 今日请求次数即将达到上限 |
| ≥ 100% | 🔴 红色 | 今日请求次数已超出限额 |

## 五、邮件通知 ✅

- **触发机制**：APScheduler 定时任务，每小时轮询一次所有用户月度使用量
- **触发条件**：月度 Token 首次达到 80% 时发送（同一自然月内只发一次，状态存 PostgreSQL）
- **收件人**：用户本人（邮箱从 AD `mail` 属性获取）
- **SMTP 配置**：写入 config.yaml，支持热加载（watchdog 监听文件变化）
- **失败处理**：SMTP 发送失败时记录日志，不标记为已发送（下次轮询会重试）

## 六、个人看板（普通用户视图）✅

### 页面布局
```
┌─────────────────────────────────────────────┐
│  月度 Token 告警进度条                        │
│  每日请求次数进度条                           │
├─────────────────────────────────────────────┤
│  累计Token │ 请求次数 │ 活跃天数 │ 日均Token  │
│  （本月 / 今日 切换）                        │
│  ⚠️ 今日视角：隐藏"活跃天数"，替换为"今日Token" │
├────────────┬─────────────┬──────────────────┤
│  趋势图    │   列表      │   模型分布        │
└────────────┴─────────────┴──────────────────┘
```

- 三个 Tab 的时间筛选**各自独立**，互不影响

### Tab 1：趋势图
- 柱状图（默认）/ 折线图（切换）
- 输入 / 输出 Token 分色叠加
- 时间筛选：最近7天 / 最近30天 / 自定义

### Tab 2：明细列表
- 字段：日期 / 模型 / 请求次数 / 输入Token / 输出Token / 总Token
- 筛选：时间范围 / 模型（多选）/ IDE类型（多选）
- 支持按列排序
- 导出 CSV：限制最近 **3 个月**（约90天），超出提示用户缩短时间范围

### Tab 3：模型分布
- 环形图：各模型 Token 占比
- 支持时间范围筛选

## 七、管理员视图 ✅

### 菜单结构
```
管理员后台
├── 用户管理        ← 用户列表 + 级别分配
├── 配额级别管理    ← 新增/编辑/删除级别（有用户时禁止删除）
├── 全局趋势        ← 所有用户汇总趋势图
├── 部门汇总        ← 按 AD department 属性汇总（取不到时显示"未知"）
└── 用量排行榜      ← 仅管理员可见，Top N（10/20/50）
```

### 7.1 用户管理
- 展示：姓名 / 域账号 / 部门（AD属性，取不到显示"未知"）/ 当前级别 / 本月Token / 今日请求次数
- 操作：修改用户所属级别

### 7.2 配额级别管理
- 展示：级别名称 / 月Token限额 / 日请求上限 / 当前人数
- 操作：**修改**各级别额度（固定 L1/L2/L3 三级，不支持新增/删除）

### 7.3 全局趋势
- 所有用户汇总每日 Token 趋势图，可按模型 / 部门分组

### 7.4 部门汇总
- 按 AD `department` 属性分组（取不到归入"未知"分组）
- 展示：部门 / 人数 / 本月Token / 本月请求次数 / 人均Token

### 7.5 用量排行榜（仅管理员）
- 按本月 Token 排序，Top N（10/20/50）
- 展示：排名 / 姓名 / 部门 / 级别 / Token用量 / 请求次数 / 配额使用率

---

> ⚠️ 排行榜为管理员专属功能，普通用户登录后不可见

## 八、技术规范 ✅

### 技术栈
| 层 | 方案 |
|----|------|
| 前端 | Vue3 + Element Plus + ECharts + Vue Router + Pinia |
| 后端 | Python FastAPI + clickhouse-driver + python-ldap + PyJWT + APScheduler + watchdog |
| 配置数据 | PostgreSQL（配额级别、用户分配、邮件发送状态） |
| 使用数据 | ClickHouse `otel.events`（只读） |
| 部署 | Docker Compose，前端 3002 / 后端 8002 |
| 访问 | 支持内网 IP 直接访问 + 域名（Nginx 反向代理） |

### 缓存策略
- ClickHouse 查询结果：5分钟 TTL（应用内存，`cachetools.TTLCache`）
- 配额级别：不缓存，每次从 PostgreSQL 直读
- config.yaml：`watchdog` 监听文件变化，变化时立即热加载

### 数据层规范
- 后端建立 `data_schema.py`，集中定义所有 ClickHouse 字段名常量
- 所有 ClickHouse 字段名使用 camelCase（对齐客户端上报格式）

### API 规范
- `GET /health` — 返回 ClickHouse / PostgreSQL / LDAP 连接状态
- 管理员操作写入后端日志（不需要前端展示页面）
- CSV 导出限制 3 个月，超出返回 400

## 九、配置文件结构

```yaml
# config.yaml

admins:
  - username: admin
    password_hash: "$2b$12$..."   # bcrypt hash，不存明文
  - username: admin2
    password_hash: "$2b$12$..."

ldap:
  server: "ldap://192.168.1.x:389"
  base_dn: "dc=company,dc=com"
  user_attr: "sAMAccountName"
  mail_attr: "mail"
  dept_attr: "department"

clickhouse:
  host: localhost
  port: 8123
  database: otel
  user: default
  password: ""

database:
  url: "postgresql://user:pass@localhost:5433/ai_usage"

smtp:
  host: ""
  port: 587
  username: ""
  password: ""
  from_name: "AI 使用助手"
  from_email: ""
```
